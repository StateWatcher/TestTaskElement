# Выборщик лотов

Сервис над каталогом лотов девелопера: публичная витрина с бронями и заявками
и админка для загрузки фида и управления наборами. Бэкенд — единый сервис
с двумя транспортами: **REST** и **JSON-RPC 2.0**; фронтенд — Vue 3 + TS + Vite +
Tailwind с переключателем транспорта.

## Запуск

```bash
cp .env.example .env   # заполнить секреты (пароль БД, ключ сессии, суперюзер)
docker compose up
```

Поднимаются три сервиса: Postgres, бэкенд и фронт.

- Витрина и админка — `http://localhost:5173`
- API напрямую — `http://localhost:8000` (Swagger `/docs`, здоровье `/health`)

Фид авто-сидингом не заливается: после старта войдите в админку
(`/admin/login`, суперюзер из `.env`), загрузите `backend/element.xml`
на экране «Наборы» и активируйте набор — до этого витрина пуста.

Локальная разработка и тесты:

```bash
# бэкенд
cd backend
python3 -m venv .venv && .venv/bin/pip install -e ".[dev]"
.venv/bin/pytest
.venv/bin/ruff check app tests

# фронтенд (нужен бэкенд на :8000 — /api проксируется Vite'ом)
cd frontend
pnpm install && pnpm dev
pnpm type-check && pnpm lint && pnpm test:unit
pnpm exec playwright install chromium && pnpm test:e2e   # при поднятом compose
```

## Устройство

```
backend/app/
  main.py               # сборка приложения, монтирует оба транспорта
  config/settings.py    # pydantic-settings: группы APP_/POSTGRES_/AUTH_
  config/logging.py     # структурные JSON-логи в stdout
  db/session.py         # engine, session factory
  models/               # SQLAlchemy ORM: Lot, LotSet, Request, Booking
  schemas/              # Pydantic DTO — общие для обоих транспортов
  repositories/         # доступ к данным, по одному на агрегат
  services/             # бизнес-логика, протокол-агностичная
    catalog.py          #   фильтры/сортировка/пагинация по активному набору
    booking.py          #   правила брони и переходы статусов лота
    requests.py         #   заявки и их статусы
    feed.py             #   парсинг XML → создание набора, активация наборов
    auth.py             #   суперюзер из конфига, подписанный токен сессии
  transports/
    deps.py             # сессия БД и cookie-авторизация — общие зависимости
    rest/routers/       # тонкие роутеры, вызывают services
    jsonrpc/dispatcher.py  # JSON-RPC 2.0: реестр методов, коды ошибок
    jsonrpc/methods.py     # привязка методов к тем же services
  domain/errors.py      # LotNotAvailable, SetNotFound, ... + enums статусов
```

Слои сверху вниз: транспорт → сервис → репозиторий → ORM. Бизнес-логика не знает
о протоколе: REST-роутер и JSON-RPC-метод — тонкие обёртки над одним сервисом,
поэтому оба транспорта гарантированно ведут себя одинаково (это закреплено тестом
паритета `test_rest_and_jsonrpc_return_same_lots`).

```
frontend/src/
  domain/               чистые типы (Lot, Booking, Application, LotSet, Page<T>) — ни Vue, ни fetch
  api/
    types.ts            интерфейс ApiClient и доменный ApiError
    http.ts             обёртка над fetch: baseURL, cookie-сессия, таймауты
    dto.ts              zod-схемы ответов, мапперы DTO → домен, сериализация запросов
    rest/, rpc/         по клиенту и разбору ошибок на транспорт
    index.ts            api() — резолвит реализацию по стору транспорта
  stores/               transport, auth, lots, bookings, applications, sets
  composables/          useLotFilters — фильтры каталога живут в query-строке
  views/                public/ (витрина), admin/ (наборы, брони, заявки, вход)
  components/ui/        shadcn-vue: код в репозитории, а не зависимость
  lib/                  notify (ApiError → тост), format, validation
frontend/nginx.conf     статика + прокси /api на backend (один origin)
```

Зеркальное правило на фронте: `views` и `stores` знают только про интерфейс
`ApiClient` и домен, а всё транспортно-специфичное заперто в `api/rest` и `api/rpc`.

### API коротко

REST (`/api`): `GET /lots`, `GET /lots/projects`, `GET /lots/{id}`,
`POST /bookings`, `POST /requests` — публичные; `POST /auth/login|logout`, `GET /auth/me`;
под сессией: `POST /sets` (multipart-загрузка фида), `GET /sets`, `POST /sets/{id}/activate`,
`GET /bookings`, `POST /bookings/{id}/cancel`, `GET /requests`, `PATCH /requests/{id}`.

JSON-RPC (`POST /api/jsonrpc`): `lots.list`, `lots.get`, `lots.projects`,
`bookings.create`, `requests.create`, `auth.login|logout|me`,
`admin.sets.upload|list|activate`, `admin.bookings.list|cancel`,
`admin.requests.list|set_status`.

Все списки (`/lots`, `/sets`, `/bookings`, `/requests` и их RPC-двойники) отдают
`Page<T>` — `items/total/page/page_size` — и принимают `page`/`page_size`.

## Отчёт

### Задача, границы, допущения, план

**Задача** — сквозной срез: PostgreSQL → бэкенд с двумя протоколами → UI,
поднимается одной командой. **Границы**: без регистрации и ролей (один суперюзер
из конфига), без CI/CD, без миграций как процесса (схема одна, создаётся на старте —
осознанное упрощение вместо Alembic; при живой эволюции схемы добавил бы его первым).

**Допущения:**

- Модель сервиса плоская: лот = квартира с названием ЖК и адресом корпуса; вся
  вложенность фида (object → buildings → flats) схлопывается при импорте. Витрине
  иерархия не нужна, фильтру по ЖК достаточно строкового поля.
- Наборы immutable: повторная загрузка фида — новый набор, а не апдейт старого.
  Публично виден только активный набор; бронировать можно только его лоты.
- Статусы фида `FREE/RESERVED/SOLD` → «в продаже / забронирован / продано».
  Отмена брони возвращает лот в «в продаже».
- Реальный `element.xml` начинается с мусорного префикса `//` перед XML-декларацией —
  парсер срезает всё до первого `<`. Битые лоты пропускаются и подсчитываются,
  а не валят импорт.
- Цена за м² не хранится: считается выражением в SQL (фильтры/сортировка) и
  `computed_field` в DTO (выдача).
- Фронт не хранит токен: бэкенд отдаёт подписанную HttpOnly-cookie, JS её не видит,
  поэтому `auth`-стор держит только флаг авторизации, а «жива ли сессия» узнаётся
  единственным способом — запросом `auth.me`.

**План**: модели и домен → сервисы → два транспорта над ними → тесты ключевой
логики → docker-упаковка → фронтенд (домен → интерфейс `ApiClient` → две реализации
→ сторы и экраны).

### REST vs JSON-RPC

Оба транспорта — тонкий слой над одними сервисами, поэтому сравнение честное:

- **REST** закладывает ресурсную модель: URL — существительное, глагол — HTTP-метод,
  статус ответа — семантика ошибки. Бесплатно достаются кэшируемость GET, Swagger
  из коробки, привычные 401/404/409, multipart для загрузки файла. Сложнее там,
  где действие не ложится на CRUD: «отменить бронь» и «активировать набор»
  пришлось оформлять POST-подресурсами (`/bookings/{id}/cancel`).
- **JSON-RPC** закладывает модель вызова процедуры: один эндпоинт, имя метода,
  типизированные params. Действия-глаголы (`admin.sets.activate`) выражаются
  естественно, протокол тривиально расширяется новым методом без дизайна URL.
  Сложнее: семантика ошибок своя (коды `-32xxx` поверх HTTP 200), нет кэширования
  и content-negotiation, файл пришлось передавать base64 (у протокола нет multipart),
  инструментальная поддержка беднее — реестр методов и валидацию params строим сами.
- Общий вывод: для публичной витрины удобнее REST, для админских операций-команд —
  RPC; при общем сервисном слое поддержка обоих стоит дёшево — новая операция
  добавляется одним методом сервиса плюс двумя тонкими привязками.

Со стороны клиента сравнение выглядит иначе, и это самое интересное наблюдение проекта:
**транспорты различаются конвертом, а не полезной нагрузкой**. Оба сериализуют одни
и те же Pydantic-схемы, поэтому тело ответа совпадает побайтово — различаются только
адресация запроса и способ доставки ошибки. Отсюда устройство `frontend/src/api`:
мапперы DTO → домен и сериализация фильтров общие (`api/dto.ts`), а транспортных
адаптеров два, и они тонкие — клиент (URL и глагол против имени метода) плюс разбор
ошибки (HTTP-статус против кода `-32xxx`). Заводить две копии одинаковых мапперов ради
симметрии значило бы платить дублированием за красивую схему папок.

### Абстракция транспорта на фронте

- **`ApiClient` абстрагирует доменные операции, а не HTTP-запросы**: `lots.list(query)`,
  `bookings.cancel(id)`, `sets.upload(file)`. Обобщённый `request(url, opts)` протащил бы
  ресурсно-глагольную семантику REST наружу и сделал бы RPC-реализацию кривой. Цена
  расширения та же, что на бэкенде: новая операция — метод в интерфейсе плюс по одной
  строке в каждом клиенте.
- **Нормализация ошибок — вторая половина абстракции.** Доменный `ApiError` с таксономией
  `validation | notFound | conflict | unauthorized | server | network`. Опорный кейс —
  бронь занятого лота: REST отвечает `409`, JSON-RPC — кодом `-32003` внутри HTTP 200,
  UI в обоих случаях показывает один тост. Это закреплено юнит-тестом
  `src/api/__tests__/errors.spec.ts` — фронтовый аналог бэкендового теста паритета.
- **Резолвер** — функция `api()`, читающая Pinia-стор транспорта; сторы зовут её в начале
  каждого экшена и никогда не держат ссылку на клиент. Осознанный минус: скрытая
  зависимость от инициализированной Pinia (в тестах — `createTestingPinia`);
  `provide/inject` чище концептуально, но setup-сторы Pinia не умеют `inject`,
  а вызовы API рождаются именно в сторах.
- **Загрузка фида** — место, где протоколы расходятся сильнее всего: 16 МБ уезжают
  multipart'ом в REST и base64 (≈22 МБ) в JSON-RPC. Это единственная операция с
  собственным таймаутом и причина `client_max_body_size 64m` в nginx.

### Конфигурация

Требование: секреты отдельно от техпараметров, смена значения — правка в одном месте.

Решение — группы настроек с env-префиксами (`pydantic-settings`): `POSTGRES_*`
(подключение к БД), `AUTH_*` (секреты авторизации), `APP_*` (технические параметры).

- **Секреты** живут только в `.env` (в git не попадает, есть `.env.example`) и
  дефолтов не имеют — без них compose и приложение не стартуют
  (`${VAR:?задайте в .env}` / обязательные поля Settings).
- **Техпараметры** имеют дефолты в коде и compose; в `.env` их выносят только
  чтобы переопределить.
- **Одно место для каждого значения**: `POSTGRES_*` из `.env` читают и контейнер
  postgres, и бэкенд — compose лишь подставляет переменную в оба сервиса. DSN
  собирается свойством `DatabaseSettings.url`, поэтому смена, например, порта БД —
  это одна строка в `.env`, без правок кода или дублирования URL.
- **Фронт не знает адреса бэкенда**: он всегда ходит в относительный `/api`, который
  в docker проксирует nginx (по имени сервиса в сети compose), а в разработке — Vite.
  Поэтому URL бэкенда нигде не продублирован, а заодно исчезает CORS: витрина и API
  живут на одном origin, и HttpOnly-cookie сессии ходит без исключений для third-party.
  `APP_CORS_ORIGINS` остаётся только для сценария «vite dev без прокси».

### Прод-культура

Структурные JSON-логи в stdout (доменные события: импорт фида, брони, логины);
осмысленные ошибки — доменные исключения транслируются в 401/404/409/422 в REST
и в коды `-32001..-32004` в JSON-RPC (соответствие между списками кодов проверено
тестом `test_error_mapping.py`, а не только держится в памяти разработчика); гонка
двойной брони закрыта атомарным `UPDATE ... WHERE status = 'in_sale'`
(`LotRepository.try_book()`) — проверка и переход состояния одним SQL-запросом,
без зависимости от блокировок конкретной СУБД, и подтверждена тестом с реальными
параллельными запросами (`test_concurrent_booking_requests_only_one_succeeds`);
cookie сессии — HttpOnly + SameSite=Lax, токен подписан (itsdangerous), пароль
сравнивается через `secrets.compare_digest`. Тесты (34) покрывают парсинг фида,
правила брони (включая конфликт через оба транспорта и реальную гонку), авторизацию,
фильтры каталога и заявки (включая паритет транспортов на админском списке).

На фронте: ноль `any` и `vue-tsc` в сборке (битые типы не доедут до образа), oxlint +
eslint + prettier, zod-валидация ответов на границе мапперов (битый ответ отвергается
там, а не течёт в компоненты), 20 юнитов на мапперы и нормализацию ошибок и Playwright-смоук,
прогоняемый **обоими транспортами**: фид → активация набора → бронь → конфликт повторной
брони и отдельно заявка → админка. Смоук окупился сразу: поймал двойной разбор zod-схемы
во вложенном наборе, который не видели ни типы, ни юниты.

### Известные ограничения фронта

- Смена активного набора в админке инвалидирует всё, что видит публичная витрина.
  Без query-кэша сброс и перезапрос делаются вручную при переходе на экран.
- Фильтр комнатности не умеет «4+»: бэкенд принимает только точный список `rooms`.
- Server state — обычные Pinia-сторы (`page/loading` на домен), но сам fetch не
  копируется в каждый стор: общий рантайм — `useAsyncResource` (`stores/asyncResource.ts`),
  счётчик поколений на ресурс отбрасывает ответ, переставший быть последним (см. ниже).
  vue-query дал бы кэш с инвалидацией по ключу, но требует разводить ключи кэша по
  транспорту — на этом объёме не окупается.

Вместо `AbortSignal` (его пришлось бы протаскивать через оба клиента ради
частного случая) каждый ресурс стора считает поколения запросов и отбрасывает ответ,
если после него уже стартовал более новый — тот же счётчик заодно убрал четыре копии
блока `loading/try/catch/finally` по сторам.
